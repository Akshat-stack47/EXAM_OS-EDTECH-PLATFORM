# STEP 1 — PROJECT UNDERSTANDING SUMMARY

## What Problem This Solves (5 Sentences)

The average Indian government exam aspirant is forced to juggle 12–18 disconnected apps, Telegram channels, YouTube playlists, and coaching institutes simultaneously, creating cognitive overload that directly harms exam performance. ExamOS kills this fragmentation by providing a single, deeply integrated ecosystem where mock test data automatically informs the AI study plan, which automatically updates the weak topic map, which feeds the personalized revision schedule. The platform solves a trust problem as much as a tech problem — aspirants constantly encounter fake PDFs, incorrect answers, and predatory monetization on existing platforms. By combining AI-verified content, a quality-gated teacher marketplace, and exam-fact-grounded AI (RAG over official syllabus data), ExamOS becomes the first platform aspirants can fully trust. The secondary problem solved is scale: government exams like SSC CGL, UPSC, and Railways release notifications simultaneously causing 20–50x traffic spikes — ExamOS must be the platform that stays online when every competitor crashes.

## Core User Journey (Day 1 → Month 12)

```
Day 1:    Signup via phone OTP → Onboarding (select exam, category, current level) → Aspirant Dashboard loads
Week 1:   First mock test → Analytics shows weak topics → AI Study Plan generated (7-day schedule)
Week 2:   Daily study sessions with flashcards + PYQs → Streak system activates → XP points accumulate
Month 1:  Re-mock → Score improvement visible → National rank tracker activates → Shares rank card on WhatsApp
Month 2:  AI Mentor activated → Personalized weak topic intervention → Study hours increase 30%
Month 3:  Upgrades to Pro plan (₹499) for full AI mentor + score prediction
Month 4:  Current Affairs engine becomes daily habit (10-min audio brief every morning)
Month 6:  Auto Form Filler saves 3 hours on exam application → Converts to loyal user
Month 9:  Joins teacher marketplace for 1-on-1 doubt sessions → Community study room participation
Month 12: Score prediction aligns with actual exam → Recommends ExamOS to 3 friends (viral loop)
```

## Single Biggest Architectural Challenge

> **The Central Student Profile Graph must be the single source of truth that all 11 modules read from and write to — in real-time — without creating a distributed data consistency nightmare.**

Every module writes different data (mock scores, study hours, news reads, form fills, mentor chats) back to one student profile. At 1M users, this creates massive write contention, cache invalidation complexity, and read-after-write consistency issues. The wrong architecture decision here will force a complete rewrite at scale. **This is the architectural bet that determines whether ExamOS survives hypergrowth.**

## 2. Backend Architecture Decision — Modular Monolith ✅

**Decision: Modular Monolith (NOT Microservices)**

| Why Modular Monolith Wins | Why Microservices Loses at This Stage |
|---|---|
| 4-6 engineers → one deploy, one codebase | 15 microservices need 15+ engineers |
| Shared Prisma transactions across modules | Distributed transactions are an unsolved problem |
| Zero inter-service network latency | Network calls between services add 10-50ms each |
| Single observability setup | Each microservice needs its own tracing, logging |
| Can extract modules to services at Phase 4 | Cannot merge microservices back into monolith |
| Faster feature development | Service versioning and API contracts slow teams down |

CREATE POLICY "profile_own_data" ON student_profiles
  FOR ALL USING (auth.uid() = user_id);

-- Test attempts: own data only
CREATE POLICY "attempts_own_data" ON test_attempts
  FOR ALL USING (auth.uid() = user_id);

-- Questions: readable by all authenticated users
CREATE POLICY "questions_read" ON questions
  FOR SELECT USING (auth.role() = 'authenticated');

-- Admins bypass RLS
CREATE POLICY "admin_bypass" ON ALL TABLES
  USING (auth.jwt()->>'role' = 'admin');
```

## 16. Security Architecture (DPDP Act 2023 Compliance)

```

1. Consent Management
   → Consent banners at signup + feature-specific consent
   → consent_logs table with timestamp + IP + what was consented

2. Data Minimization
   → Only collect what's necessary for the feature
   → Aadhaar collected ONLY for form filler (not required for core)

   → DELETE /api/user/me endpoint (soft-delete first, hard-delete 30 days)
   → Cascade deletion across all tables (Prisma onDelete: Cascade)
   → Remove from Pinecone vectors, Redis cache, S3 files

4. Data Localization
   → ALL data stored in AWS Mumbai (ap-south-1) region
   → No cross-border data transfer

5. Breach Notification
   → Automated: if >100 failed auth attempts → PagerDuty alert → 72h notification window

6. Aadhaar Vault Security
   → Encrypted with AES-256 (AWS KMS managed key)
   → Key rotation every 90 days
   → Access only via authenticated API (no direct DB access to document_vault)
   → Audit log: every access to document_vault logged with user_id + timestamp + reason

7. PII Fields in DB
   → Encrypted at field level: phone, name, dob, category
   → Hashed for indexing: phone_hash for lookups

# STEP 7 — TECH STACK FINALIZATION

| Layer | Recommended | Why | Alternative | Tradeoff |
|---|---|---|---|---|
| **Frontend Web** | Next.js 14 (App Router) | SSR for SEO (exam searches), RSC for performance | Vite + React | Vite has no SSR built-in; exam pages need SEO |
| **Frontend Mobile** | React Native + Expo | Shared codebase with web hooks/types, OTA updates | Flutter | Flutter means separate codebase, less code sharing |
| **UI Library** | TailwindCSS + shadcn/ui | Rapid development, customizable, no runtime CSS | MUI, Chakra | MUI forces their design language, harder to customize |
| **State Management** | Zustand + React Query | Zustand for UI state, RQ for server state (cache + sync) | Redux Toolkit | Redux is verbose, RQ makes it unnecessary |
| **Backend API** | Node.js + Express (TypeScript) | Same language as frontend = shared types | FastAPI (Python) | Python is great for AI but slower for REST API |
| **AI Service** | FastAPI (Python, separate) | Python ecosystem for AI (LangChain, LlamaIndex) | Node.js AI | Node.js LLM libraries are immature |
| **Database** | Supabase (PostgreSQL) + Prisma | Managed DB + Auth + Realtime + RLS in one | Self-managed RDS | Self-managed needs DevOps FTE we don't have |
| **Cache** | Redis (Upstash in Phase 1) | BullMQ + sessions + leaderboards + rate limits | Memcached | Memcached has no pub/sub, no sorted sets |
| **Vector DB** | Pinecone | Managed, easy to start, Hindi text support | Weaviate | Weaviate self-hosted = more DevOps burden |
| **Queue** | BullMQ (Redis-backed) | TypeScript-native, great DX, retry/backoff built-in | AWS SQS | SQS has no UI, harder to debug (use SQS at Phase 3) |
| **AI/LLM** | Claude API (Haiku for simple, Sonnet for complex) | Better at Indian exam content, lower hallucination rate | GPT-4o | GPT-4o fallback for resilience |
| **File Storage** | AWS S3 + CloudFront | Industry standard, cheap, CDN native | Cloudflare R2 | R2 is cheaper but ecosystem less mature |
| **Auth** | Supabase Auth | OTP-native, JWT, RLS integration, free | Firebase Auth | Firebase = another vendor, Supabase already chosen |
| **Payments** | Razorpay | India-specific, all payment methods, UPI, GST invoicing | Stripe | Stripe India support is incomplete |
| **Email** | Resend + React Email | Developer-friendly, beautiful templates, free tier | SendGrid | SendGrid pricing is expensive at scale |
| **SMS/OTP** | MSG91 | India-specific, cheapest OTP rates, DLT registered | Twilio | Twilio 5x more expensive in India |
| **Push Notifications** | FCM (Firebase Cloud Messaging) | Free, reliable, works for both iOS + Android | OneSignal | OneSignal adds another vendor for no gain |
| **Search (Phase 1)** | Supabase pg_trgm | Already in Supabase, no new service, good enough | Elasticsearch | Elasticsearch overkill + expensive for Phase 1 |
| **Search (Phase 3)** | Typesense | Open-source, fast, typo-tolerant, 10x cheaper than Elastic | Elasticsearch | Elasticsearch requires dedicated cluster |
| **Deployment (Phase 1)** | Railway.app | Zero DevOps, automatic deploys, $5/month to start | Vercel | Vercel has no persistent Node.js backend |
| **Deployment (Phase 2+)** | AWS ECS Fargate | No Kubernetes management, auto-scaling, AWS native | Kubernetes | K8s requires dedicated DevOps engineer |
| **Observability** | Sentry (errors) + Pino (logs) + Better Uptime | Free tier covers Phase 1-2 | Datadog | Datadog $50+/month per host |
| **Secrets** | Doppler | Free tier, team-friendly, injects into all environments | AWS Secrets Manager | AWS SM is complex to set up for small teams |
| **CI/CD** | GitHub Actions | Free for public repos, tight GitHub integration | CircleCI | GitHub Actions is sufficient and free |
| **Testing** | Vitest + Playwright + k6 | Modern, fast, TypeScript-native testing stack | Jest + Cypress | Jest is slower, Cypress is more complex |

### 🔴 Wrong Technology Choices in Original Blueprints

| Wrong Choice | Why It's Wrong | Correct Choice |
|---|---|---|
| MongoDB alongside PostgreSQL | PostgreSQL JSONB handles everything MongoDB does; 2 DBs = 2x operational burden | PostgreSQL only |
| ClickHouse from Day 1 | Requires dedicated DBA + BI team; 0 users = no analytics to run | Supabase + read replica |
| Elasticsearch from Day 1 | pg_trgm handles search up to 100M rows; ES requires separate cluster management | pg_trgm → Typesense at Phase 3 |
| Firebase Realtime alongside Supabase | Two real-time systems = data sync nightmare | Supabase Realtime only |
| 15 Microservices from Day 1 | Impossible for 4-6 person team | Modular Monolith |
| GraphQL + gRPC + REST simultaneously | Three API paradigms = unmaintainable | REST only (OpenAPI spec) |
| Kubernetes from Day 1 | Requires full-time DevOps; overkill at 0 users | Railway → ECS → EKS (graduated) |
| Go for high-throughput services | Requires Go expertise; Node.js handles Phase 1-2 easily | Node.js → Go extraction if needed |
| Agora video in Phase 1 | Complex integration, licensing cost, not needed until marketplace | Phase 2+ |
| LLM Fine-tuning (Mistral 7B) in early phases | Requires GPU budget + ML expertise + 6+ months training data | RAG-first, fine-tune at Phase 4 |

# STEP 10 — FINAL SUMMARY OUTPUT

## 1. FINAL RECOMMENDED ARCHITECTURE

ExamOS v2.0 should be built as a **TypeScript-first Modular Monolith** on Node.js, backed by **Supabase (PostgreSQL + RLS + Realtime)** as the single source of truth, with a **separate Python FastAPI service for AI inference** and **BullMQ on Redis for all async operations**. The frontend is a **Next.js 14 App Router** web app and **React Native + Expo** mobile app sharing types and business logic via a Turborepo monorepo. This architecture supports confident growth from 0 to 100K users without rewrite, and allows progressive extraction of services into microservices only when specific performance bottlenecks justify it.
